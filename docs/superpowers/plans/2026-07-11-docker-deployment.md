# Docker Deployment Setup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a production Docker image + `docker-compose.yml` so the app can run on the VPS's PHP 8.4 (via container) instead of the VPS's system PHP 8.2, with a persistent queue worker and persistent SQLite/storage data, exposing a plain HTTP port for the VPS's existing Apache to reverse-proxy to.

**Architecture:** A multi-stage `Dockerfile` (Node build stage → Composer vendor stage → `php:8.4-fpm-alpine` runtime running nginx + php-fpm via supervisord) produces one image, used by two `docker-compose.yml` services (`app` serving HTTP, `queue` running `queue:work`) sharing named volumes for the SQLite file and `storage/`.

**Tech Stack:** Docker, Docker Compose, Alpine Linux, nginx, supervisord, PHP 8.4-fpm.

## Global Constraints

- Local dev is unaffected — no docker-compose override for local dev, per user's explicit choice.
- The SQLite file is mounted as a volume **at the file path**, never at the `database/` directory (that directory also ships `migrations/`, `factories/`, `seeders/` from the image, which a directory-level volume would shadow).
- `APP_KEY` is generated once and kept stable across restarts — it must never be auto-regenerated by the entrypoint.
- No Apache config, TLS, or CI/CD is added — those are out of scope per the approved spec.
- **This sandbox has no `docker` binary available** (verified: `which docker` fails), so tasks in this plan cannot be verified with a real `docker build`/`docker compose up` here. Each task's "verification" step is a careful manual read-through instead of an automated test run. Task 4 gives the user the exact commands to run for real verification on a machine that has Docker (their VPS, or any local Docker install).

### Task 1: `Dockerfile` and `.dockerignore`

**Files:**
- Create: `Dockerfile`
- Create: `.dockerignore`

**Interfaces:**
- Consumes: `docker/nginx.conf`, `docker/supervisord.conf`, `docker/entrypoint.sh` (created in Task 2 — referenced by `COPY`/`ENTRYPOINT` instructions here, so this task's Dockerfile will reference paths that don't exist until Task 2 completes; that's fine, this task is not build-verified until Task 3).
- Produces: an image buildable as `docker build -t presence-rotary .` — consumed by `docker-compose.yml` in Task 3.

- [ ] **Step 1: Create `.dockerignore`**

```
.git
.gitignore
.claude
.worktrees
docs/superpowers
node_modules
vendor
tests
storage/logs/*
storage/framework/cache/*
storage/framework/sessions/*
storage/framework/views/*
public/build
database/database.sqlite
.env
.env.*
!.env.example
*.md
```

- [ ] **Step 2: Create the `Dockerfile`**

```dockerfile
# syntax=docker/dockerfile:1

FROM node:22-alpine AS frontend
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM composer:2 AS vendor
WORKDIR /app
COPY composer.json composer.lock ./
RUN composer install --no-dev --optimize-autoloader --no-interaction --no-scripts --no-progress

FROM php:8.4-fpm-alpine AS runtime

RUN apk add --no-cache \
        nginx \
        supervisor \
        sqlite \
        icu-dev \
        libzip-dev \
        libpng-dev \
        oniguruma-dev \
    && docker-php-ext-install \
        pdo_sqlite \
        mbstring \
        dom \
        gd \
        zip \
        bcmath \
        intl \
        pcntl \
        opcache \
        xml \
        fileinfo

WORKDIR /var/www/html

COPY . .
COPY --from=vendor /app/vendor ./vendor
COPY --from=frontend /app/public/build ./public/build

RUN mkdir -p storage/framework/{cache,sessions,views} storage/logs \
    && chown -R www-data:www-data storage bootstrap/cache \
    && chmod -R 775 storage bootstrap/cache

COPY docker/nginx.conf /etc/nginx/http.d/default.conf
COPY docker/supervisord.conf /etc/supervisor/conf.d/supervisord.conf
COPY docker/entrypoint.sh /usr/local/bin/entrypoint.sh
RUN chmod +x /usr/local/bin/entrypoint.sh

EXPOSE 80

ENTRYPOINT ["/usr/local/bin/entrypoint.sh"]
CMD ["supervisord", "-c", "/etc/supervisor/conf.d/supervisord.conf", "-n"]
```

- [ ] **Step 3: Manual review**

Read the file back and confirm: each `COPY --from=` stage name matches a
`FROM ... AS <name>` above it (`vendor`, `frontend`); the final stage has no
name (implicit last stage, correct since it's the one `docker build`
produces by default); extensions list matches the spec's recommendation
(`pdo_sqlite`, `mbstring`, `gd`, `zip`, `bcmath`, `intl`, `pcntl`, `opcache`,
`xml`, `fileinfo`, plus `dom` which ships built into PHP's core on Alpine
and doesn't need `docker-php-ext-install` — remove it from the
`docker-php-ext-install` list if a build later shows it's already
compiled-in; this is exactly the kind of thing Task 3's real `docker build`
will surface that a read-through can't).

- [ ] **Step 4: Commit**

```bash
git add Dockerfile .dockerignore
git commit -m "feat: add production Dockerfile"
```

### Task 2: nginx, supervisord, and entrypoint config

**Files:**
- Create: `docker/nginx.conf`
- Create: `docker/supervisord.conf`
- Create: `docker/entrypoint.sh`

**Interfaces:**
- Consumes: nothing new.
- Produces: the three files `COPY`'d by Task 1's `Dockerfile` — Task 1's `docker build` (run for real in Task 3) will fail immediately if any of these are missing or malformed, which is the actual verification for this task.

- [ ] **Step 1: Create `docker/nginx.conf`**

```nginx
server {
    listen 80;
    server_name _;
    root /var/www/html/public;
    index index.php;

    client_max_body_size 20M;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        fastcgi_pass 127.0.0.1:9000;
        fastcgi_index index.php;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        include fastcgi_params;
    }

    location ~ /\. {
        deny all;
    }
}
```

- [ ] **Step 2: Create `docker/supervisord.conf`**

```ini
[supervisord]
nodaemon=true
user=root

[program:php-fpm]
command=php-fpm --nodaemonize
autostart=true
autorestart=true
stdout_logfile=/dev/stdout
stdout_logfile_maxbytes=0
stderr_logfile=/dev/stderr
stderr_logfile_maxbytes=0

[program:nginx]
command=nginx -g "daemon off;"
autostart=true
autorestart=true
stdout_logfile=/dev/stdout
stdout_logfile_maxbytes=0
stderr_logfile=/dev/stderr
stderr_logfile_maxbytes=0
```

- [ ] **Step 3: Create `docker/entrypoint.sh`**

```sh
#!/bin/sh
set -e

if [ ! -f database/database.sqlite ]; then
    touch database/database.sqlite
fi

php artisan migrate --force
php artisan config:cache
php artisan route:cache
php artisan view:cache

exec "$@"
```

- [ ] **Step 4: Make it executable and commit**

```bash
chmod +x docker/entrypoint.sh
git add docker/nginx.conf docker/supervisord.conf docker/entrypoint.sh
git commit -m "feat: add nginx, supervisord, and entrypoint config for Docker runtime"
```

### Task 3: `docker-compose.yml`

**Files:**
- Create: `docker-compose.yml`

**Interfaces:**
- Consumes: the `Dockerfile` from Task 1 (via `build: .`).
- Produces: nothing consumed by a later task in this plan — Task 4 uses this file to run real verification.

- [ ] **Step 1: Create `docker-compose.yml`**

```yaml
services:
  app:
    build: .
    restart: unless-stopped
    env_file: .env
    ports:
      - "${APP_PORT:-8080}:80"
    volumes:
      - sqlite-data:/var/www/html/database/database.sqlite
      - storage-data:/var/www/html/storage

  queue:
    build: .
    restart: unless-stopped
    env_file: .env
    command: php artisan queue:work --tries=3 --max-time=3600
    volumes:
      - sqlite-data:/var/www/html/database/database.sqlite
      - storage-data:/var/www/html/storage

volumes:
  sqlite-data:
  storage-data:
```

- [ ] **Step 2: Commit**

```bash
git add docker-compose.yml
git commit -m "feat: add docker-compose.yml for app and queue-worker services"
```

### Task 4: Real verification (requires a machine with Docker) + handoff notes

**Files:** none (verification + documentation only)

This sandbox has no `docker` binary, so this task cannot be run by the
assistant. Hand these steps to the user (or run them yourself on the VPS or
any machine with Docker installed):

- [ ] **Step 1: Prepare the `.env` file**

On the target machine, in the repo root:

```bash
cp .env.example .env
docker compose run --rm app php artisan key:generate --show
```

Paste the generated key into `.env` as `APP_KEY=...`. Set `APP_URL`,
`ADMIN_EMAIL`, `ADMIN_PASSWORD`, `MAIL_*`, and `APP_PORT` (the free port on
the VPS — the user confirmed 8080 is already taken there, so pick another
before deploying) as needed.

- [ ] **Step 2: Validate the compose file**

Run: `docker compose config`
Expected: prints the fully-resolved config with no errors (confirms YAML
is valid and `env_file`/`${APP_PORT}` interpolation resolves).

- [ ] **Step 3: Build the image**

Run: `docker compose build`
Expected: all three stages (`frontend`, `vendor`, runtime) complete without
error. This is where any Dockerfile mistakes Task 1's manual review missed
(e.g. an extension that's already built-in and errors on
`docker-php-ext-install`) will actually surface — fix and rebuild as
needed.

- [ ] **Step 4: Start the stack and confirm migrations ran**

Run: `docker compose up -d`
Then: `docker compose logs app | grep -i migrat`
Expected: migration output, no errors, both `app` and `queue` containers
show as running (`docker compose ps`).

- [ ] **Step 5: Confirm the app responds**

Run: `curl -I http://127.0.0.1:${APP_PORT:-8080}/`
Expected: `HTTP/1.1 200 OK` (the public attendance form).

- [ ] **Step 6: Confirm the queue worker actually processes jobs**

Log in as admin through the exposed port, close a session with the
thank-you email checkbox checked (as in the earlier manual verification of
that feature), then:

Run: `docker compose logs queue --tail=50`
Expected: log output showing the `AttendanceThankYouMail` job was
processed (no exceptions).

- [ ] **Step 7: Point Apache at the container**

This step is not part of the repo (VPS-specific Apache config). Example
vhost snippet to hand to the user, using whatever port they picked for
`APP_PORT`:

```apache
<VirtualHost *:80>
    ServerName presence.example.com

    ProxyPreserveHost On
    ProxyPass / http://127.0.0.1:8080/
    ProxyPassReverse / http://127.0.0.1:8080/
</VirtualHost>
```

(Requires `a2enmod proxy proxy_http` on Debian/Ubuntu Apache, or the
equivalent module enablement. TLS termination, if any, stays configured on
this same Apache vhost exactly as it is for the VPS's other sites — out of
scope for this repo.)

- [ ] **Step 8: Report back**

Confirm to the user which steps passed, and fix forward for anything that
didn't (most likely candidates: a PHP extension that's actually built into
the `php:8.4-fpm-alpine` base and errors when `docker-php-ext-install`
tries to install it again, or an nginx/php-fpm socket mismatch).
